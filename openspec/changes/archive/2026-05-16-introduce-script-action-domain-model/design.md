## Context

当前项目已经有平台无关的 `InputDevice` 端口、fake device 和 macOS adapter，但核心层仍以固定 demo workflow 表达业务流程。用户实际拥有的是多个拼装好的脚本，每个脚本由已经编排好的点击、拖拽、等待命令组成，并且只需要先支持执行一次。

这次设计的重点是把“脚本是什么”从 demo 函数中抽出来，形成核心领域模型。坐标相关概念也需要尽早明确：第一版脚本只绑定一个窗口，脚本内所有点击和拖拽动作默认都在这个窗口内执行；窗口可以是整个屏幕，也可以是指定区域；区域由矩形记录范围，点表示动作位置。

## Goals / Non-Goals

**Goals:**

- 定义 `Script`、`Action`、`Click`、`Drag`、`Wait` 的核心领域模型。
- 定义 `Window`、`ScreenWindow`、`AreaWindow`、`Rect`、`Point` 的基础几何模型。
- 让 `Script` 绑定一个窗口，并让脚本内点击和拖拽通过该窗口解析出可交给 `InputDevice` 的屏幕坐标。
- 让脚本运行一次时按顺序执行所有动作。
- 将输入设备端口调整为跨平台点击、拖拽语义，不把鼠标按键暴露为领域概念。
- 保持 adapter 不依赖脚本领域对象，只接收已经解析后的输入设备操作。

**Non-Goals:**

- 不实现循环、条件判断、失败恢复、图像识别或 OCR。
- 不实现脚本文件格式、脚本持久化或 Repository。
- 不实现自动查找游戏窗口，也不处理标题栏、边框、缩放等平台细节。
- 不支持单个脚本内混用多个窗口；多窗口编排留到后续能力。
- 不把 `Move` 作为第一版脚本动作；如果未来需要鼠标移入触发 UI，优先建模为 `Hover`。
- 不引入 `MouseButton`；手机端和触控设备没有左键、右键、中键这种领域语义。

## Decisions

### 使用显式动作类型表达脚本命令

`Script` 持有有序动作集合，动作先限定为 `Click`、`Drag`、`Wait`。每种动作用独立类型表达，而不是用字符串命令和字典参数表达。

理由：独立类型能让领域语义更清楚，也更容易测试。后续增加动作时可以扩展模型，而不是在一个松散命令结构里堆字段。

备选方案：使用统一 `Command(name, args)`。这个方案短期更灵活，但会把领域约束推迟到运行时，不利于学习和维护领域模型。

### 将窗口绑定在 Script 层级

第一版脚本只在一个窗口内执行。`Script` 持有一个窗口，脚本内所有 `Click` 和 `Drag` 动作都默认使用这个窗口解析自己的 `Point`。`Click` 和 `Drag` 不再单独记录窗口。

理由：用户当前描述的是“一个脚本在一个窗口里执行一次”，不是跨多个窗口编排。把窗口放在 `Script` 层级可以减少每个动作的重复字段，也能明确一个脚本的坐标上下文。

备选方案：让每个动作单独绑定窗口。这个方案更灵活，但会让第一版模型变复杂，也会暗示一个脚本可以混用多个窗口。后续如果确实需要跨窗口脚本，可以再把窗口提升为动作级覆盖项。

### 明确坐标解析规则

`Point(x, y)` 表示脚本窗口内的像素点。`ScreenWindow` 把 `Point` 原样解析为屏幕坐标；`AreaWindow(Rect(left, top, width, height))` 把 `Point(x, y)` 解析为 `Point(left + x, top + y)`。

坐标解析不做边界夹取。也就是说，如果 `Point` 落在 `AreaWindow` 的 `Rect` 之外，解析规则仍然按偏移计算，是否拒绝越界点留给后续能力决定。

理由：第一版先定义最小、可预测的坐标换算。边界校验、百分比坐标、屏幕缩放和内容区修正都属于更高阶规则，过早加入会让核心模型变宽。

备选方案：在 `AreaWindow` 中强制校验点必须落在矩形范围内。这个方案更严格，但游戏脚本调试阶段可能需要临时点到区域边界外；并且不同平台对窗口边框、内容区的解释还没稳定。

### 保留现有 InputDevice 作为执行端口

脚本 runner 负责把领域动作转换成 `InputDevice` 调用。`InputDevice` 和平台 adapter 不需要认识 `Script`、`Action`、`Window`。

理由：这保持了现有 port-and-adapter 边界。核心脚本模型可以用 fake device 测试，平台 adapter 继续只处理鼠标指针操作。

备选方案：让 adapter 直接执行领域动作。这样会让平台实现依赖业务模型，后续多个 adapter 会重复解释脚本语义。

### 第一版不把 Move 建模为脚本动作

脚本动作只包含 `Click`、`Drag`、`Wait`。`Move` 不进入领域模型，因为多数游戏脚本里的移动鼠标只是执行点击或拖拽前的底层准备，不是用户想表达的业务动作。

理由：领域动作应该表达脚本意图。点击、拖拽、等待会改变流程或等待流程变化；单纯移动指针通常不产生游戏行为。当前 macOS adapter 可以继续在拖拽实现内部移动到起点，但这是 adapter 细节。

备选方案：加入 `Move`。这个方案会把桌面鼠标的底层操作提前暴露到跨平台脚本模型里，对手机端和触控端不自然。后续如果确实需要“鼠标移入后触发 UI”的语义，应单独引入 `Hover(window, point, duration)`。

### 第一版不引入 MouseButton

点击和拖拽表达跨平台的触控/指针动作，不包含左键、右键、中键等鼠标按键概念。如果某个平台底层需要鼠标左键，应该由 adapter 内部映射。

理由：目标是跨平台游戏脚本，手机端、模拟器和触控设备没有右键、中键语义。把 `MouseButton` 放入领域模型会让桌面鼠标细节污染通用脚本语言。

备选方案：保留 `MouseButton` 并默认为左键。这个方案兼容现有 demo，但会增加用户暂时不需要的概念，也会让后续触控 adapter 需要解释无意义的按键参数。

### 第一版只支持像素点，不引入百分比点

本 change 只根据用户指定的 `Point` 和 `Rect` 建立像素坐标模型。百分比点是合理的后续扩展，但暂不进入第一版。

理由：当前请求明确提到 `point` 和 `rect`，先把窗口、点和动作执行边界建稳。百分比会引入比例、取整和边界夹取规则，适合作为后续单独变更。

备选方案：同时加入 `RatioPoint`。这会让第一版规格变宽，测试维度也增加。

## Risks / Trade-offs

- 坐标系语义被误解 → 在规格和测试里覆盖 `ScreenWindow` 与 `AreaWindow` 的坐标解析差异。
- `ScreenWindow` 缺少真实屏幕尺寸 → 第一版可以把屏幕窗口视为无偏移坐标系，不负责边界校验。
- 指定区域和游戏内容区域不完全一致 → 第一版由调用方传入准确 `Rect`，自动窗口发现和内容区修正留到后续 adapter 能力。
- 单个脚本暂不支持多个窗口 → 当前脚本模型更简单；后续多窗口需求出现时，再引入动作级窗口覆盖或更高层编排。
- 动作模型过早扩展 → 本次只保留点击、拖拽、等待，避免把条件、循环、识别等未来能力混进第一版。
- 旧的 `Coordinate`、`PointerTiming`、`MouseButton` 与新模型重复 → 实现时用 `Point`、动作持续时间和跨平台点击/拖拽语义替代，避免同一概念存在两套名字。
