## 背景

仓库目前还没有应用代码结构，因此这个变更可以从一个小型 Python-first 布局开始，不需要迁移既有脚本。目标是在编写可复用游戏自动化脚本的同时练习架构能力。核心架构约束是：玩法和业务逻辑不能直接调用 macOS API，平台差异必须放在适配器实现背后。

初始平台目标是 macOS，用 Python 模拟鼠标指针移动、点击和拖动。未来可以增加其他桌面平台、移动端自动化或远程控制后端，而不修改脚本核心层。

## 目标 / 非目标

**目标：**

- 建立一个平台无关的输入设备接口，用于表达鼠标指针动作。
- 把业务流程保留在核心层，并且只依赖该接口。
- 提供一个 macOS Python 鼠标指针适配器，支持移动、点击和拖动。
- 提供一个 demo workflow，证明核心层既可以连接 macOS adapter，也可以连接测试替身。
- 保持框架足够小，适合用来练习和理解架构分层。

**非目标：**

- 不实现计算机视觉、屏幕识别、OCR、游戏状态解析或路径规划。
- 不在初始版本实现键盘模拟，只为未来输入操作预留扩展空间。
- 初始实现不支持 Windows、Linux、移动端或浏览器自动化。
- 不承诺兼容所有游戏客户端，也不绕过操作系统或应用的安全限制。
- 不构建生产级 bot 调度器、插件系统或 GUI。

## 技术决策

### 使用 port-and-adapter 结构

业务脚本放在 core package 中，并接收一个类似 `InputDevice` 的接口。平台实现放在 adapter 模块中，先从 macOS 鼠标指针适配器开始。

理由：这样可以明确依赖方向。核心逻辑依赖稳定端口，适配器依赖操作系统相关库。核心 workflow 可以用 fake device 测试，后续也可以继续增加新的平台 adapter。

备选方案：把辅助函数直接写在脚本里。这样做单个 demo 更快，但会继续把平台行为和 workflow 决策混在一起，违背主要目标。

### 先把指针操作建模为同步命令

初始接口暴露简单同步操作，例如带坐标和时间参数的 `move_to`、`click` 和 `drag_to`。

理由：同步命令 API 更容易理解和测试，也足够支撑鼠标 demo。之后仍然可以在更高层 workflow 对象中包装这些动作。

备选方案：一开始就使用异步或事件命令队列。这对复杂自动化可能有价值，但会在核心边界验证前过早引入并发决策。

### 在边界上保持坐标表达平台无关

核心层传递明确的屏幕坐标，以及简单的按钮和持续时间参数。底层库需要的任何转换都由 macOS adapter 负责。

理由：核心层需要一套小而稳定的设备动作语言。显式处理坐标可以避免与 macOS API 产生隐藏耦合，同时仍然支持可运行 demo。

备选方案：使用丰富的平台专用事件对象。这会把 adapter 细节暴露给业务逻辑，也会让后续平台扩展更困难。

### 为 macOS 自动化使用窄依赖包装

macOS adapter 会把选定的自动化库或原生集成隔离在单个模块内。如果权限或依赖不可用，adapter 会抛出清晰的 setup/runtime 错误，而不是让底层错误泄漏到核心 workflow。

理由：macOS 鼠标指针控制依赖本机权限和外部包。包装层可以把这些运行环境细节隔离在脚本和测试之外。

备选方案：在 demo 脚本中直接调用自动化库。这样可以验证鼠标控制，但验证不了架构边界。

## 风险 / 取舍

- macOS 辅助功能权限可能阻止鼠标模拟 -> 记录 setup 要求，并暴露清晰的 adapter 错误。
- 屏幕坐标会受到显示缩放和多显示器配置影响 -> 保持 demo 简单，并明确坐标假设。
- 窄接口在加入键盘、截屏或手势后可能需要演进 -> 先聚焦 pointer port，后续通过新需求扩展，而不是现在过度泛化。
- 真实游戏可能拒绝合成输入，或需要时序和状态反馈 -> 将这个变更定位为框架 demo，不承诺兼容具体游戏。

## 迁移计划

当前没有既有应用代码，因此不需要迁移。实现时应逐步新增 package 结构、demo 和测试。回滚方式是删除新增 package、demo、依赖声明和测试。

## 待确认问题

- 实现时应该优先选择哪种 macOS 自动化后端：`pyautogui` 这样的高层 Python 库、基于 Quartz 的实现，还是先在 adapter 背后做后端选择？
- 键盘输入是否应该在 pointer demo 后立即加入，还是等鼠标指针边界验证完成后作为后续 capability 提出？
